# APP-ARCHITECTURE.md

## 문서 목적

이 문서는 PRZM의 초기 앱 구조를 실무 구현 수준에서 정의한다.

목적은 아직 기술 스택이나 세부 라이브러리를 확정하지 않은 상태에서도, 어떤 라우트가 존재하고 어떤 모듈이 어떤 책임을 가지며, 화면 상태와 데이터 흐름이 어디에서 관리되는지를 명확히 하는 것이다.

PRZM은 지도 화면이 중심인 제품이지만, 이후 확장을 위해 라우트와 기능 모듈 구조를 함께 가져가야 한다. 따라서 앱 구조는 "여러 라우트가 있지만 지도 화면이 메인인 구조"를 기준으로 설계한다.

## 아키텍처 원칙

- 메인 경험은 지도 화면에서 시작하고 끝난다.
- 라우트는 얇게 유지하고, 실제 책임은 기능 모듈이 가진다.
- 기능 경계는 `지도`, `필터`, `요약`, `상세 분석`, `예측`, `데이터` 기준으로 나눈다.
- 전역 상태는 사용자의 탐색 맥락을 유지하는 데 필요한 최소 범위만 가진다.
- 현재 상태와 예측 상태는 같은 선택 컨텍스트를 공유해야 한다.
- 초기에는 단순한 구조를 우선하되, 전국/전 연령 확장을 막지 않는 수준의 일반성은 유지한다.

## 앱 구조 개요

초기 PRZM 앱은 아래 두 층으로 이해할 수 있다.

1. 라우트 계층
사용자가 접근하는 화면 진입점을 정의한다.

2. 기능 모듈 계층
실제 제품 경험과 상태, 데이터, UI 조합 책임을 가진다.

즉, 라우트는 페이지를 보여주는 진입점이고, 제품 로직은 기능 모듈이 담당한다.

## 추천 라우트 구조

초기 MVP에서 라우트는 많지 않아야 한다.

### 1. `/`

메인 지도 화면이다.

이 라우트가 PRZM의 핵심 제품 경험을 담당한다.

포함 기능:

- 지도
- 지역/질병/연령 선택
- 요약 카드
- 상세 패널 또는 드로어
- 분석
- 예측

### 2. `/about` 또는 `/method`

선택적 보조 라우트다.

서비스 소개, 데이터 기준, 예측 해석 방식 같은 보조 정보가 필요할 경우 사용한다.

초기 MVP에서는 필수가 아니지만, 메인 경험과 분리된 설명성 콘텐츠를 담는 데 유용하다.

### 3. `/unsupported` 또는 공통 상태 화면

지원 범위 밖 지역이나 기능 제한 상황을 설명하는 보조 진입점으로 둘 수 있다.

다만 초기에는 별도 라우트보다 공통 상태 컴포넌트로 처리하는 편이 더 단순할 수 있다.

## 추천 폴더 구조

아래 구조는 특정 프레임워크에 종속되지 않는 개념적 기준이지만, React 계열 SPA 또는 SSR 앱 구조에 그대로 적용하기 쉽다.

```text
src/
  app/
    routes/
    providers/
    store/
    layout/
  entities/
    disease/
    region/
    observation/
    forecast/
  features/
    map/
    filters/
    summary/
    analysis/
    forecast/
    selection-context/
  widgets/
    top-control-bar/
    summary-card/
    detail-panel/
    bottom-drawer/
  shared/
    ui/
    lib/
    config/
    types/
    constants/
  pages/
    home/
    about/
```

## 계층별 책임

## 1. `app/`

앱 전역 설정과 부트스트랩 계층이다.

### 책임

- 전역 provider 연결
- 앱 라우트 정의
- 전역 store 연결
- 최상위 레이아웃 구성

### 포함 가능 항목

- 라우터 설정
- 데이터 클라이언트 설정
- 전역 테마 및 환경 구성

## 2. `entities/`

도메인 데이터 단위를 정의하는 계층이다.

### 책임

- 핵심 엔터티 타입 정의
- 엔터티 단위 selector 또는 mapper
- 원시 데이터 구조와 화면용 구조 사이의 최소 변환

### 주요 엔터티

- `disease`
- `region`
- `observation`
- `forecast`

이 계층은 제품 기능보다는 데이터 개념을 중심으로 한다.

## 3. `features/`

실제 제품 경험을 구성하는 핵심 계층이다.

각 기능 모듈은 자신의 상태, 데이터 조합, UI 조립 책임을 가진다.

### `features/map`

#### 책임

- 지도 표시
- 지역 선택
- 위험도 레이어 표시
- 질병 확산 오버레이 표시

### `features/filters`

#### 책임

- 지역 선택 UI
- 질병 선택 UI
- 연령 선택 UI
- 선택 변경 액션 처리

### `features/summary`

#### 책임

- 현재 선택 상태 기반 요약 계산
- 요약 카드 렌더링
- 현재 위험 및 변화 방향 해석

### `features/analysis`

#### 책임

- 추세 차트
- 확산 속도 변화
- 연령/성별 분포 표시

### `features/forecast`

#### 책임

- 일주일 예측 표시
- 한달 예측 표시
- 방향 해석과 신뢰도 노출

### `features/selection-context`

#### 책임

- 현재 선택 상태 관리
- 지도, 요약, 분석, 예측이 같은 맥락을 공유하도록 연결

이 모듈은 초기 PRZM에서 매우 중요하다. 사용자가 선택한 `지역 / 질병 / 연령 / 기준 시점`을 앱 전체 경험에서 일관되게 유지시켜야 하기 때문이다.

## 4. `widgets/`

여러 feature를 조합해 만든 화면 단위 UI 블록 계층이다.

### 책임

- 상단 제어 바
- 요약 카드
- 상세 패널
- 모바일 하단 드로어

`features`가 로직 중심이라면, `widgets`는 화면 조립 중심이다.

## 5. `shared/`

앱 전역에서 재사용되는 공통 요소 계층이다.

### 책임

- 버튼, 카드, 칩 같은 공용 UI
- 날짜/숫자 포맷 유틸
- 공통 타입
- 상수와 환경값

여기에는 제품 의미가 강한 로직을 넣지 않는다.

## 6. `pages/`

라우트에 직접 연결되는 페이지 진입점 계층이다.

### 책임

- 라우트별 화면 조립
- 필요한 widget과 feature 연결

초기에는 `home`이 가장 중요한 페이지다.

## 메인 라우트 구성 방식

`/` 라우트는 아래와 같은 조립 구조를 가진다.

```text
Home Page
  -> Top Control Bar
  -> Map View
  -> Summary Card
  -> Detail Side Panel / Bottom Drawer
      -> Analysis Section
      -> Forecast Section
```

핵심은 `Home Page`가 모든 로직을 직접 가지지 않는다는 점이다.

페이지는 조립만 하고, 실제 상태와 데이터 책임은 feature 계층이 가진다.

## 상태 구조

초기 PRZM의 상태는 아래처럼 계층화하는 것이 적절하다.

## 1. 전역 탐색 상태

앱 전체 경험에서 공유되어야 하는 상태다.

### 포함 항목

- 선택 지역
- 선택 질병
- 선택 연령
- 현재 기준 시점
- 상세 패널 열림 여부

### 이유

지도, 요약 카드, 상세 분석, 예측이 모두 같은 선택 맥락을 공유해야 하기 때문이다.

## 2. 기능 로컬 상태

특정 기능 안에서만 필요한 상태다.

### 예시

- 드롭다운 열림 여부
- 차트 탭 선택 상태
- 모바일 드로어 확장 단계

이 상태는 전역으로 올리지 않는 것이 맞다.

## 3. 서버/원격 데이터 상태

외부 데이터 로딩 결과와 캐시 상태다.

### 예시

- 지역별 관측 데이터 로딩 상태
- 질병별 추세 데이터 로딩 상태
- 예측 데이터 로딩 상태

이 상태는 일반 UI 상태와 분리해 다루는 것이 유지보수에 유리하다.

## 데이터 흐름

초기 PRZM의 기본 데이터 흐름은 아래와 같다.

1. 앱 진입 시 기본 지역 범위 데이터 로드
2. 지도에 위험도와 기본 요약 표시
3. 사용자가 지역/질병/연령을 선택
4. 선택 컨텍스트 갱신
5. 선택 컨텍스트에 맞는 관측 데이터와 예측 데이터 재조회 또는 재계산
6. 지도, 요약, 분석, 예측 UI 동시 갱신

핵심은 하나의 선택 컨텍스트가 여러 기능 모듈을 동시에 갱신한다는 점이다.

## 선택 컨텍스트 모델

초기 앱 구조에서는 아래 형태의 공통 선택 컨텍스트를 상정하는 것이 적절하다.

```text
selectionContext
  - regionId
  - diseaseId
  - age
  - snapshotId
  - panelState
```

이 컨텍스트는 지도, 요약, 상세 분석, 예측의 공통 입력 역할을 한다.

## 반응형 구조 책임

PRZM은 데스크톱과 모바일에서 정보 구조는 같고 표현 방식만 다르다.

따라서 반응형 책임은 아래처럼 나누는 것이 좋다.

- `features`는 데이터와 상태 책임을 공유한다.
- `widgets`가 데스크톱/모바일 표현 차이를 흡수한다.
- 예를 들어 상세 정보는 같은 데이터와 섹션을 쓰되, 데스크톱에서는 `Detail Side Panel`, 모바일에서는 `Bottom Detail Drawer`로 표현한다.

이 방식은 로직 중복을 줄인다.

## API/데이터 경계

초기 앱 구조에서 API 계층은 아래 원칙을 따르는 것이 적절하다.

- 엔터티 단위로 읽기 쉬운 fetch 함수 또는 query 단위를 둔다.
- UI는 원시 API 응답을 직접 소비하지 않는다.
- 관측 데이터와 예측 데이터는 분리된 경계로 읽되, 같은 selection context를 입력으로 사용한다.

예시 경계:

- `getRegionObservations`
- `getDiseaseSummary`
- `getObservationBreakdown`
- `getForecast`

## 오류 및 빈 상태 처리 구조

오류와 빈 상태는 화면 부수 효과가 아니라 아키텍처 레벨에서 고려해야 한다.

### 원칙

- 지도, 요약, 상세, 예측은 각각 로딩/오류/빈 상태를 가질 수 있다.
- 전체 페이지를 막는 치명적 오류와 섹션 수준 오류를 구분한다.
- 데이터 부족 상태는 오류가 아니라 정상적인 상태로 다룬다.

## 확장 원칙

이 구조는 아래 확장을 감당할 수 있어야 한다.

### 1. 지역 확장

서울/경기도에서 전국으로 확장되어도 `region`과 `map` feature 구조는 유지되어야 한다.

### 2. 연령 확장

어린이 중심에서 전 연령으로 확장되어도 `selection-context`와 `breakdown` 처리 방식이 유지되어야 한다.

### 3. 보조 라우트 확장

향후 `/about`, `/method`, `/alerts` 같은 보조 라우트가 추가되어도 메인 로직은 여전히 feature 계층에 머물러야 한다.

### 4. 비교 기능 확장

향후 비교 기능이 추가되면 selection context를 단일 선택에서 다중 선택으로 확장할 수 있어야 한다. 다만 초기 MVP에서는 단일 선택만 다룬다.

## 초기 범위에서 하지 않을 것

- 페이지별로 중복된 상태를 따로 관리하지 않는다.
- 모든 로직을 `pages/home`에 몰아넣지 않는다.
- 차트/지도 컴포넌트가 직접 원시 API 응답을 해석하게 하지 않는다.
- 전국 확장을 미리 과설계하지 않는다.

## 구현 시작 순서 제안

초기 구현은 아래 순서로 진행하는 것이 적절하다.

1. `selection-context`와 `home` 페이지 골격 구성
2. `map` feature와 기본 위험도 표시
3. `filters`와 요약 카드 연결
4. `detail-panel` 또는 `bottom-drawer` 구조 연결
5. `analysis`와 `forecast` feature 연결
6. 로딩/빈 상태 정리

이 순서는 PRZM의 핵심 사용자 흐름과 직접 맞물린다.
