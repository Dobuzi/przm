# TECH-STACK.md

## 문서 목적

이 문서는 PRZM의 초기 구현에 적합한 기술 스택을 정의한다.

목적은 단순히 도구 목록을 나열하는 것이 아니라, 현재 PRZM의 제품 구조와 앱 아키텍처에 맞는 기술 선택을 정리하고, 왜 그 선택이 적절한지 기록하는 것이다.

초기 PRZM은 지도 중심의 클라이언트 인터랙션, 공유 선택 상태, 원격 데이터 기반 분석, 반응형 패널/드로어 UI가 핵심이다. 따라서 기술 선택 역시 이 문제를 가장 단순하고 견고하게 해결하는 방향이어야 한다.

## 기술 선택 원칙

- MVP에 필요한 복잡도만 도입한다.
- 지도와 데이터 표현이 핵심이므로, 클라이언트 인터랙션에 강한 스택을 우선한다.
- UI 상태와 서버 상태는 분리해서 다룬다.
- 과도한 풀스택 프레임워크나 무거운 상태 관리 도구는 피한다.
- 초기 생산성과 이후 확장성을 함께 고려한다.
- 한국 지역 데이터와 지도 오버레이 표현에 불리한 선택은 피한다.

## 추천 스택 요약

초기 PRZM에 대한 추천 스택은 아래와 같다.

- 프레임워크: `React + Vite`
- 언어: `TypeScript`
- 라우팅: `React Router`
- 서버 상태: `TanStack Query`
- 앱 UI 상태: `Zustand`
- 지도: `Mapbox`
- 차트/시각화: `Recharts` 또는 `visx`
- 스타일링: `Tailwind CSS`
- 품질 도구: `ESLint`, `Prettier`, `Vitest`

이 조합은 현재 PRZM의 문서 구조, UI 방향, 앱 구조와 가장 잘 맞는다.

## 프론트엔드 프레임워크

### 선택

`React + Vite`

### 이유

- PRZM의 핵심은 서버 렌더링보다 클라이언트 중심 지도 인터랙션이다.
- 빠른 개발 서버와 단순한 프로젝트 구조가 초기 MVP에 유리하다.
- `features` 중심 아키텍처를 깔끔하게 구현하기 쉽다.
- React 생태계에서 지도, 상태, 차트, 스타일링 선택지가 넓다.

### 대안 비교

#### Next.js

장점:

- 향후 SEO, 콘텐츠 페이지, 서버 기능 확장에 유리하다.

단점:

- 현재 PRZM의 핵심 가치에 비해 프레임워크 복잡도가 높다.
- 초기 MVP 단계에서는 SSR 이점보다 클라이언트 인터랙션 비중이 훨씬 크다.

### 결론

지금 단계에서는 `React + Vite`가 더 단순하고 적절하다.

## 언어

### 선택

`TypeScript`

### 이유

- PRZM은 `질병`, `지역`, `관측값`, `예측`, `선택 컨텍스트`처럼 명확한 도메인 구조를 가진다.
- 데이터 구조가 화면, 차트, 지도, 예측에 동시에 연결되기 때문에 타입 안정성이 중요하다.
- 기능 모듈 경계와 엔터티 계층을 유지하기 쉽다.

### 결론

JavaScript보다 `TypeScript`가 확실히 적합하다.

## 라우팅

### 선택

`React Router`

### 이유

- PRZM은 여러 라우트가 있지만 메인 경험은 `/` 중심이다.
- 초기 라우트 구조는 단순하고, 라우트는 얇게 유지할 계획이다.
- `React + Vite` 조합에서 가장 무난하고 검증된 선택이다.

### 사용 방향

- `/` 메인 지도 화면
- `/about` 또는 `/method` 같은 보조 설명 페이지

### 결론

초기에는 `React Router`면 충분하다.

## 상태 관리

PRZM은 상태를 두 종류로 나눠 다루는 것이 맞다.

1. 앱 UI 상태
2. 서버/원격 데이터 상태

이 둘을 같은 도구로 억지로 해결하지 않는 것이 중요하다.

## 앱 UI 상태

### 선택

`Zustand`

### 이유

- `지역 / 질병 / 연령 / 기준 시점 / 패널 열림 상태` 같은 선택 컨텍스트를 여러 feature가 공유한다.
- Context만으로 처리하면 곧 앱 전반에 얽힐 가능성이 높다.
- Redux 계열 도구는 현재 단계에 비해 과하다.
- Zustand는 단순한 전역 상태를 작게 유지하기 좋다.

### 사용 대상

- 선택 컨텍스트
- 패널/드로어 열림 상태
- 일시적 앱 UI 상태

## 서버 상태

### 선택

`TanStack Query`

### 이유

- 관측 데이터와 예측 데이터는 원격 데이터이며 캐시, 재조회, 로딩 상태가 중요하다.
- 선택 컨텍스트가 바뀔 때 재조회와 캐시 전략을 자연스럽게 가져갈 수 있다.
- 로딩/오류/빈 상태를 일관되게 다루기 좋다.

### 사용 대상

- 지역별 관측 데이터
- 질병 요약 데이터
- 상세 분석 데이터
- 예측 데이터

### 결론

`Zustand + TanStack Query` 조합이 가장 적합하다.

## 지도

### 선택

`Mapbox`

### 이유

- PRZM은 지도 위에서 위험도 색상, 확산 범위, 선택 상태 등 커스텀 레이어 표현이 중요하다.
- 단순한 핀 지도보다 데이터 시각화 중심 지도에 가깝다.
- React와 함께 쓰기 좋고, 스타일 제어와 레이어 표현 유연성이 높다.

### 대안 비교

#### Google Maps

장점:

- 사용자에게 익숙하다.

단점:

- 데이터 오버레이와 커스텀 시각화 측면에서 상대적으로 제한적일 수 있다.

#### Naver Maps

장점:

- 한국 사용자 맥락에 익숙하다.

단점:

- 현재 프론트엔드 생태계와 확장성 측면에서는 `Mapbox`가 더 유연하다.

### 결론

초기 PRZM의 지도 핵심 경험에는 `Mapbox`가 가장 적합하다.

## 차트와 시각화

### 선택

초기 추천은 `Recharts`다.

향후 더 정교한 커스터마이징이 필요하면 `visx`를 검토한다.

### 이유

- 초기 MVP에서 필요한 차트는 추세 차트와 간단한 분포 차트 중심이다.
- 너무 무거운 데이터 시각화 프레임워크는 과하다.
- 빠르게 구성하고 읽기 쉬운 차트를 만들기에는 `Recharts`가 유리하다.

### 대안 비교

#### visx

장점:

- 커스터마이징 자유도가 높다.

단점:

- 초기 MVP 구현 속도는 떨어질 수 있다.

### 결론

초기에는 `Recharts`, 필요 시 `visx` 재검토가 적절하다.

## 스타일링

### 선택

`Tailwind CSS`

### 이유

- 지도 중심 UI는 간결한 레이아웃 조정과 반응형 제어가 매우 중요하다.
- 카드, 패널, 드로어, 오버레이 같은 UI를 빠르게 반복 구현하기 좋다.
- 초기 디자인 토큰과 컴포넌트 구조를 가볍게 올리기 좋다.

### 대안 비교

#### CSS Modules

장점:

- 파일 단위 격리가 명확하다.

단점:

- 레이아웃 반복 작업과 상태별 스타일 조합이 상대적으로 번거롭다.

#### styled-components / emotion

장점:

- 컴포넌트와 스타일을 강하게 결합할 수 있다.

단점:

- 현재 단계에서는 Tailwind보다 단순성과 속도 면에서 이점이 약하다.

### 결론

초기 MVP에는 `Tailwind CSS`가 가장 실용적이다.

## UI 컴포넌트 방식

### 선택

외부 대형 UI 프레임워크를 중심에 두지 않고, `shared/ui`에 PRZM 맞춤형 컴포넌트를 직접 쌓는 방식을 추천한다.

### 이유

- PRZM은 일반 관리도구 UI보다 지도 중심 커스텀 레이아웃 비중이 높다.
- 디자인 시스템 규모가 아직 크지 않다.
- 범용 UI 키트를 많이 들이면 오히려 구조가 무거워질 수 있다.

### 결론

필요한 저수준 primitive만 쓰고, 핵심 컴포넌트는 직접 구성하는 편이 적절하다.

## 품질 도구

### 선택

- `ESLint`
- `Prettier`
- `Vitest`
- `Testing Library`

### 이유

- 초기 단계부터 코드 품질과 기본 테스트 구조를 갖추는 것이 좋다.
- PRZM은 상태와 데이터 조합이 많아질 수 있으므로, 최소한의 테스트 기반이 필요하다.

### 테스트 우선순위

- selection context
- 요약 계산 로직
- 예측 결과 표시 로직
- 주요 화면 상태 렌더링

## 데이터 계층

### 선택 방향

- API 클라이언트 또는 fetch 계층은 `shared/lib` 또는 `entities` 근처에 둔다.
- 화면은 원시 API 응답을 직접 소비하지 않는다.
- 엔터티 단위 변환 함수 또는 selector를 둔다.

### 이유

- `관측 데이터`와 `예측 데이터`를 화면이 직접 해석하면 결합도가 높아진다.
- 앱 구조 문서에서 정의한 `entities + features` 계층과 잘 맞아야 한다.

## 환경 변수 및 설정

초기에는 아래 종류의 설정이 필요할 가능성이 높다.

- Mapbox 토큰
- API base URL
- 데이터 스냅샷 또는 개발 모드 설정

이 값들은 `shared/config` 또는 `app/config`에 모으는 것이 적절하다.

## 향후 재검토 포인트

아래 항목은 초기 MVP 이후 다시 검토할 수 있다.

### 1. 프레임워크 재검토

- 콘텐츠 페이지와 SEO 중요도가 커질 경우 `Next.js` 재검토 가능

### 2. 차트 라이브러리 재검토

- 커스텀 시각화 요구가 커지면 `visx` 또는 더 낮은 수준 도구 검토

### 3. 지도 재검토

- 한국 지역 데이터, 비용, 스타일링 요구에 따라 지도 벤더 재평가 가능

### 4. 상태 관리 재검토

- 비교 기능, 다중 선택, 복잡한 워크플로우가 생기면 상태 구조 재정비 필요

## 결론

초기 PRZM에는 아래 스택이 가장 적절하다.

- `React + Vite`
- `TypeScript`
- `React Router`
- `Zustand`
- `TanStack Query`
- `Mapbox`
- `Recharts`
- `Tailwind CSS`
- `ESLint`, `Prettier`, `Vitest`

이 조합은 현재 PRZM의 목표인 지도 중심 모니터링, 선택 컨텍스트 공유, 상세 분석, 방향성 예측 UI를 가장 단순하게 구현할 수 있는 선택이다.
